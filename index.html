1.              Document Overview

The purpose of this document is to provide a guide to accessing and using the dentsu WhatsApp platform. It is assumed that all users have a basic knowledge of WhatsApp for business including template types and guidelines.

 

2.              UI Access

Access to the platform is facilitated by dentsu, using the dentsu secure portal. This will require any user accessing the platform to login using 2 factor authentication (Push notification via okta App or Authentication code).

 

Please contact your account manager to request access to the platform. Once authorised, an email will be sent to the user inviting them to setup an account.

 

Access is available by the following URL : https://myapps.dentsu.com/

3.1.          Template Management

The Template Management portal allows a user to create, edit and delete WhatsApp templates. The table shows a complete list of the templates available, with the template name, the language, its current status and the category type.

Note: You may also create/ edit and delete templates using the Meta Console.

A search bar is provided at the top, which will dynamically update the table as text is input. You can also filter by language using the top right drop down box.

 

 

 

 

 

 

 

 

 

 

 

3.2.          Create/Edit template

Note: A template may only be edited once every 24 hours!

All templates created will be submitted to Meta for approval, this takes a few seconds. If Meta decides that the template type has been defined incorrectly, it will be automatically changed. Typically, this is from Utility to Marketing.

 

Name: Give your template a unique name, an ID is automatically generated

Language: Select the language for the template. You will need to create a new language variant for each template you wish to create

Category: Select a category type, this can be either Marketing, Utility or Authentication

Header Type: Select either text or image. If you select an image an additional field will be displayed to upload a sample image for approval.

Note: Header images shown here may not reflect a live experience The header templates used are sent at the time of send in real time and can be defined in the Chatbot/Journey editor or the rule definition file itself (more information below).

Body Text: This is the core part of the template, enter your text here. You can add parameters here as well, if you want to pass in additional information into the template. For example, a Name or Order information. The UI will allow you to add up to 3 parameters, if you need to add more than 3, please use the Meta console.

Parameter / Sample Values: If you have added parameters in the Body text above, please enter the parameter name and a sample value here. The sample values will be used for template approval once submitted to Meta.

Buttons: Select a button type if you wish to add a button the template:

Custom - Enter text to be displayed on the button

Quick Reply most common this also the user to quickly select a predefined response (e.g. see more options)

Visit Website select this option to add a URL

Complete Flow You can add this button to enable the user to complete a WhatsApp Flow. A drop down list is provided to select the Flow to link to this button

Copy Offer Code Enables the copying of an offer code included in the template

Note: there are limitations on URL tracking within WhatsApp depending on the region and business headquarters. Please contact your account team if you are unsure.

Once you have created/edited a template, click publish to save changes and submit to Meta for approval.

3.4.          Journeys / Chatbots

The UI currently allows a non-technical user to create and manage templates. To create a journey of templates/ messages a configuration file will need to be edited to inform the platform what to do when a message arrives or an interaction event occurs on a template / message.

The configuration file is formatted in JSON and should be managed by a technical user accessed via an S3 bucket. The file is formed of the following components:

3.4.1.     Rules

This component contains an array of rule objects. Each rule defines conditions for triggering a specific action. Each rule contains the following components:

3.4.1.1.                    Trigger

Specifies the type of event that triggers the rule:

        type: An array defining the type of trigger

o   Allowed values: text, button or interactive

        Template(optional): The template associated with an interactive or button trigger

o   Allowed values: template or interactive name

        allowedTime(optional): defines the time constraints when the trigger can be activated

o   Allowed values: days (days of the week the trigger is valid), time_range: (time range the trigger is valid

o   Example: {"days": ["Monday", "Tuesday", "Wednesday","Thursday","Friday"], "time_range": "19:00-23:59"}

3.4.1.2.                    checkType

Specifies how the values should be compared. Possible values:

        equals: Exact match required

        contains: Partial match allowed

3.4.1.3.                    values

An array of values to match against the trigger

3.4.1.4.                    caseCheck

Indicates whether case sensitivity is considered (Y for yes, N for no).

3.4.1.5.                    action

Specifies the action(s) to take if the rule conditions are met. Multiple actions can be defined for a single trigger.

        type: Defines the action type (e.g., template, db, MediaCard).

If type is template:

o   templateName: The name of the template to be sent

o   headerParameters: If header is present in template we have to provide it otherwise we dont provide:

  type: Specifies the content type (e.g., image, text).

  image: Contains the link to the image.

o   bodyParameters: Parameters for the message body.

o   flowParameters: Additional parameters related to flow handling.

If type is db:

  databaseType: Specifies the database type (dynamodb, mysql).

  itemStructure: Defines the key-value pairs to be inserted into the database.

  tableName: The table where the data should be written.

If type is MediaCard:

  TemplateName, headerParameters, bodyParameters, FlowParameters remain same as template action.

  catalogueID (required): ID of the catalog to query products from.

  catalogueFilter: Filters catalog entries.

o   Uses flexible filtering (e.g., i_contains) to filter out product on different attributes in catalogue. You can use response variables in the filter string

o   Eg :- "Fashion>{{flow_response.Department}}>{{flow_response.Product_Category}}"

  catalogueReturn: Array of fields to return from the catalog (e.g., ["id", "name", "price"]).

  limit: Maximum number of items to return. If template supports 3 cards we limit 3. if limit is less than cards template support we will loop through the result that will duplicate some cards to fulfil the quota.

 

3.4.1.6.                    defaultTemplate

Defines the default response template if no rules match.

o   type: Specifies the response type (e.g., template)

o   templateName: The name of the fallback template

3.4.2.     Example Rule

{

"trigger": {

"type": ["button"],

"template": "clr_indoor_outdoor"

},

"checkType": "equals",

"values": ["Interior"],

"caseCheck": "N",

"action": [

{

"type": "template",

"templateName": "clr_room_v2",

"headerParameters": [

{

"type": "image",

"image": {

"link": "https://example.com/image.jpg"

}

}

],

"bodyParameters": [],

"flowParameters": ["button"]

},

{

"type": "db",

"databaseType": "dynamodb",

"itemStructure": {

"mobile_number": "{{from_number}}",

"optout_datetime": "{{current_datetime}}"

},

"tableName": "{{table.Optout}}"

},

{

"type": "db",

"databaseType": "mysql",

"itemStructure": {

"user_id": "{{user_id}}",

"action_time": "{{current_datetime}}"

},

"tableName": "user_actions"

}

]

}

This rule triggers when a button associated with the clr_indoor_outdoor template is clicked, and the value is Interior. If matched, it:

1. Sends the clr_room_v2 template with an image.

2. Writes an opt-out entry into the DynamoDB table {{table.Optout}}.

3. Records the action in a MySQL table user_actions.

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

3.5.          Conversations Portal Live Chat UI

The Live Chat UI or Conversations portal allows users to talk directly to customers via WhatsApp.

 

 

This portal allows a user to have a conversation with a customer that has interacted with the WhatsApp chat including any pictures or documents that have been sent or received.

 

Note: that you can only send a message within 24 hours of the customers last message, as per Meta Policy, this is called a Customer Service Window.

 

 

The left pane displays a list of conversations, click on any of these numbers to view the conversation in the middle pane, and details about the customer that have been captured in the right-hand pane

 

A user can also search for a number and filter numbers by assigned, unassigned or by Agent.

The middle pane is the chat window, this will display all the interactions with the customer. To respond to a message, type a message in the chat bar and click send. You can also click on a message received by a customer to reply in context.

 

To upload a document, such as a PDF or image, click the + icon in the chat bar. This will add the document to the chat in draft, a user can then click send or add a message and click send.

 

Note: that you can only send a message within 24 hours of the customers last message, as per Meta Policy.

 

The right-hand pane displays information captured about the customer in the bottom half and the ability to assign an Agent (Consultant) in the top half.

 

To assign an Agent, click the drop down and select an Agent from the list.

To re-assign an Agent, select a new Agent from the drop-down list.

 

To update the status, select a status from the drop-down list.

 

New Agents can be added as part of the User Access request detailed above, please let the team know that a new user will be an Agent when requesting access.

 

 

 

 

 

 

 

 

3.6.          Campaign Management

Campaign Management allows WhatsApp messages to be sent to bulk to customers in the form of Marketing or Utility templates. This is done by using a file in this first release, Integrations to other platforms will be released as when required.

NB: The Campaign currently allows the sending of messages to 1 template only.

3.6.1.     Campaign List

Lists the all campaigns have been setup, allows for the editing of the campaign details and the setup itself

 

3.6.2.     Create Campaign

Create campaign page allows a user to create the campaign, enter metadata and select the template that will need to be sent.

        Campaign Name: Give the campaign a unique name

        Campaign Description: A description for the campaign being setup

        Template Name: A drop down list of approved templates to use in this campaign. (As above the campaign functionality supports 1 template only.

        Template Language: will be auto populated based on the template selected. User can then select the language to use for this campaign

        Template Parameters: will be auto populated if there are parameters defined in the template selected above

 

Once you click Save this campaign will now be available to setup in the Campaign List:

3.6.3.     Campaign Setup

The setup page allows the user to configure when the campaign should run:

        CampaignID: Auto generated

        Campaign Name: defined by the user when the campaign is initially created

        Campaign File Name: The name of the file that must be used

        Campaign File Header: Indicates any additional fields that must be included in the file for any parameters

        Upload: Click this button to upload the file of customers you want to broadcast to

        Run: Select now or a select a date time from the date picker

A visual editor is currently in development to allow the creation of WhatsApp Flows in a drag and drop style canvas. In the meantime, Flows can be created using the Meta Console.
